# Neovim Config

This directory contains the primary Neovim configuration for the dotfiles repository. It mixes legacy Vim runtime reuse with a mostly Fennel-authored Neovim config, compiled Lua artifacts, lazy-loaded plugins, and a small amount of checked-in local data.

## Note for LLM

Keep this file updated whenever top-level details change.
If you are not sure about something, leave a note for other developers to review.

### Editing principles

- Start from `init.lua` to understand boot order.
- Prefer editing `fnl/**` and `ftplugin/*.fnl` when a matching source file exists.
- Most files under `lua/**` and `ftplugin/*.lua` are generated by `nfnl`; many contain `do not edit` headers.
- `.ignore` excludes `lua/**/*.lua` from search results, so searches may only show the Fennel source.
- `init.lua` prepends `~/.vim` to `runtimepath` and sources `~/.vimrc`, so some behavior still lives in the repo-root Vim config.
- `lazy-lock.json` is the plugin lockfile and should be updated intentionally, not casually reformatted.

### Known caveats

- Not every compiled Lua file currently has a tracked Fennel source. `lua/plugins/numb.lua` and `lua/config/lsp.lua` look like compiled artifacts but do not have matching `fnl/` files in this repo.
- The checked-in `spell` entry is currently a broken symlink in the repo tree. Shared spell files appear to live under repo-root `.vim/spell`, and `init.sh` also links that root spell directory into the installed Neovim config.

## Startup Flow

- `init.lua` disables several builtin Vim plugins, enables truecolor, bootstraps `lazy.nvim` if needed, enables the Lua loader, sets leader keys, then sources repo-root `~/.vimrc`.
- After the legacy Vim config is sourced, Neovim plugin setup is delegated to `require("lazy").setup("plugins")`.
- `lua/plugins.lua` is a small hand-authored bootstrap file that registers `lazy.nvim` itself and the `Olical/nfnl` plugin.
- The `nfnl` plugin loads `require("config")`, which comes from the compiled `lua/config/init.lua` generated from `fnl/config/init.fnl`.

## Functional Layout

- Global editor behavior lives in `fnl/config/init.fnl` and `fnl/config/util.fnl`.
- `fnl/config/util.fnl` is the main helper module for keymaps, buffer-local mappings, visual-range helpers, word extraction, and common LSP attach behavior.
- `fnl/plugins/*.fnl` defines lazy.nvim plugin specs one plugin or feature area at a time.
- Filetype-specific behavior lives under `ftplugin/`, with Clojure receiving the heaviest custom treatment.
- Completion is centered on `nvim-cmp`, `snippy`, LSP, Conjure, buffer/path/spell sources, and optional Codeium.
- Search/navigation is centered on Telescope with recent-files, file-browser, undo, live grep, and git pickers.
- LSP setup includes at least Fennel, Clojure, Java, Go, Kotlin, TypeScript, Emmet, and LTEX wiring.
- Theme handling is dynamic and includes auto dark-mode switching plus Neovide font/transparency settings.

## Language-Specific Notes

- Clojure config detects `pubspec.yaml`, `project.clj`, or `deps.edn` to choose a REPL startup command.
- Clojure mappings also include Conjure/shadow-cljs helpers and uncommenting utilities.
- LTEX stores its custom dictionary in `data/ltex/ltex.dictionary.en-US.txt`.
- `data/plenary/filetypes/ext.lua` maps `.cljd` files to the `clojure` filetype.
- `basic.fnl` conditionally enables `codeium.nvim` only when `OPENAI_API_KEY` is present in the environment.

## Validation Notes

- There is no standalone test suite for this config.
- Reasonable validation is to open Neovim, let lazy.nvim load, and check representative workflows with `:Lazy`, `:checkhealth`, and the relevant filetypes.
- After changing Fennel sources, verify that the compiled Lua mirrors are still in sync before considering the edit complete.
